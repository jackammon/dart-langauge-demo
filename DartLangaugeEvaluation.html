Dart’s History, Google’s Answer for Large Scale App Development

At the inception of the internet, websites served the function of hosting links and static content, namely academic and scientific writing. The exciting part of the early internet was the ability to access information from anywhere in an instant. As the internet continued to grow websites continued to improve however they did not compare to native desktop applications. In 2007 the iPhone was released to the world resulting in a large shift. Mobile applications were now capable of everything websites were and because of the technological advantage of being a native application they could provide dynamic interactivity, greatly increasing the user experience. This consequently increased the market's expectation for dynamic interactivity across all applications, web or not. JavaScript, the current standard for web development, was just inadequate for the task and a solution was desperately needed.

In 2011 Google announced their new language Dart, developed by Gilad Bracha and Lars Bak for “structured web programming.”  In the keynote announcement presentation, Dart was presented as the solution for making the development of large scale applications easier [1]. During the announcement the presenters stated three design goals for Dart:

“1. Create a structured yet flexible language for web programming.

  2. Make Dart feel familiar and natural to programmers and thus easy to learn.

  3. Ensure that Dart delivers high performance on all modern web browsers and environments ranging from small handheld devices to server-side execution“ [2].

During the presentation the presenters shared that JavaScript is not the competition, fragmented mobile platforms are. In the immediate next slide of the presentation, both presenters talked about their issues with large scale application development in JavaScript and then proceeded to present Dart’s language features as solutions. Dart was announced with the ability to compile to a number of different platforms from a single code base. At the very end the presenters also announced that Google had plans to integrate a Dart VM into Chrome.

 

That last part of the announcement received a lot of criticism for fracturing the web. Many drew comparisons to Microsoft's VBScript from the 90s browser wars. They put VBScript in Internet Explorer, giving developers the option to not develop their client-side code in JavaScript if wanted. However VBScript was only available within Internet Explorer and divided the internet, removing a standard for web browsers [3].  This was not received well by the web development community as many thought Google sought to outright overtake and replace JavaScript, even though Google was vocal that this was not the case.

Dart was by no means the only solution for large structured web applications in the market. Throughout the early 2010s several other solutions sprang up with web frameworks Angular in 2010, React in 2013, and Vue in 2014 and the language TypeScript in 2012. These frameworks however are built upon either JavaScript or TypeScript and largely sought to play within the confines JavaScript established, Dart sought to break from that paradigm completely and create a language optimized for building modern enterprise-level user interfaces [4]. Google has the desire to develop web-applications with the same level of experience as native mobile applications and Dart is at the core of their whole mobile and web-application ecosystem. Google has a multi-platform UI framework built upon Dart called Flutter. Flutter consists of both an SDK and a UI component library, however, what makes Flutter powerful is that a single code base compiles natively to iOS, Android and the web.

Interestingly, Google is also the largest contributor to Angular, another multi-platform application framework which is currently far more popular than Dart. Angular however is built upon TypeScript, another solution for the JavaScript problem. TypeScript was announced a year after Dart as “a language for application-scale JavaScript development” [5], a very similar announcement goal to Dart without the intent (expressed or not) to replace JavaScript. In a 2012 blog post Google shared that they announced Dart to fix some of the issues with large scale web application development, namely better tooling and static type annotations. In that same blog post Google said that “TypeScript has validated both of these statements” [4]. Statements like these, along with Google’s heavy support of Angular show their awareness that the market is motivated far more by the economical factors, not just the technological.

This is why Google has its hand in two very different approaches to solve modern application development. On the surface, Dart is a fantastic technological solution for building modern user interfaces for applications, however, whether or not we see the rise of Dart is to be determined. Personally, there needs to be another shift in the market demand for mobile development unification for Dart to become a major player outside the walls of Google as so many large scale projects have already been built upon languages and frameworks like Angular, React, and Swift. That being said, Dart does not have to be widely adopted outside of Google for it to be a success. The goal was to simplify the development of enterprise-scale applications and if Dart achieves that for a company the size of Google it has done its job. Google has some of the best UI/UX of any major player. Google’s applications have an incredibly consistent user experience across platforms. In Github’s 2019 State of the Octoverse report, Dart was the found to be the fastest-growing language with 532% growth over the past year, with Flutter also climbing to #2 in open source projects by contributors [6]. Whichever way Dart goes, Google has made their claim for the future of app development and only time will tell if it will be widely adopted.

Language Overview

One of the primary design goals for Dart was to have a structured yet flexible language. To bring needed structure to web application development Dart was designed around the C-style, object-oriented paradigm. Straight from the language specification it says that Dart is a “pure object-oriented programming language” [7:9]. This means that in Dart, everything is an object. Dart is class-based, adding structure and also implements mixin-based inheritance which adds modularity and flexibility. Another design goal was immediate productivity for Google’s developers [1]. This is evidenced in Dart’s simple and pragmatic syntax and semantics, there are no surprises. Examples of syntactic elements will be shown in the following sections.

One of the knocks against JavaScript is that it often has unpredictable behavior and a primary source of this is the result of implicit type conversion at runtime. JavaScript’s weak typing is to blame for the ambiguity that creates this behavior. Google avoids this entirely by strongly typing Dart, in the process adding some needed structure. One of the language goals was to have both structure and flexibility though. Finding this happy middle can be a challenge. Google’s solution for flexibly with a strongly typed language is to also include dynamic typing and optional static-type annotations, which was largely influenced by Strongtalk’s optional static typing system [8]. Types can be determined at runtime through a type inference system. Dart’s style guide encourages avoiding type annotating when the types can be inferred [9]. Similar to Java, Dart also supports generic types through what they call a dynamic type. Dart’s strong yet dynamic and optional typing creates the structure lacking in JavaScript without making huge flexibility sacrifices.

Data Types

Dart has eight built-in data types including numbers, strings, booleans, lists (arrays), sets, maps, runes, and symbols. These data types are familiar for those who have used any C-like language. Dart is a pure object-oriented language, so every variable in Dart is an object.

Numbers objects in Dart can be either an int or a double (64-bit floating-point) and are subtypes of a num object. Within the num object the reside the basic operators +, -, /, and * and a few other operations such as abs() are found. Similar to Python there is a provided math library that can be imported for more functionally.

Similar to C and Java, Dart Strings are immutable sequences of UTF 16 code units, however, to support 32-bit Unicode values Dart, along with Google’s other language Go have something called Runes [10]. When C was created, strings were created as an array of pointers to chars, which are 1 byte (8-bit values). At the time 1 byte was enough to represent all 128 ASCII values, and who could ever need more than 128 characters? Well today Unicode has over 150,000 characters [11]. To solve this problem Dart strings are sequences of bytes and sometimes runes as some characters take more than a single byte to represent.

Collections

Darts core collections include Lists, Sets, and Maps and function as one would expect. Lists are essentially simple arrays that hold objects and maintain order by a 0-based index. Sets also hold objects however do not maintain the notion of order and each item has to be unique. Maps are a classic dictionary, mapping key-value pairs. While these collections are nothing unfamiliar several of Dart’s language features are specifically included to make building interfaces easier. One interesting thing is that Maps can be declared using a traditional constructor much like Java and JavaScript, however, they can also be declared using syntax identical to creating JavaScript objects. This syntactic sugar is nice when working with JSON.

Tradition map construction syntax

var user = Map<String, String>();

user['username'] = 'bsnider';
user['password'] = '5f4dcc3b5aa765d61d8327deb882cf99';

JSON style map construction syntax

var user = {
   'username': 'bsnider',
   'password': '5f4dcc3b5aa765d61d8327deb882cf99',
};

Another nice language feature when working with collections is Dart’s collection if. These allow for easy building of collections that vary depending on the application state. Below is an example where collection if is used to build out the items of a navigation bar [12].

var navBar = [
   'Home',
   'Treading',
   if (loggedIn) 'Personal Feed',
   if (loggedIn) 'Sign Out',
   if (!loggedIn) 'Sign In'
];

Functions

As already mentioned, Dart is a pure object-oriented language which means that everything in Dart is an object and this includes functions. Also importantly in Dart variables only hold object references. When functions are treated like any other variable they are called first-class functions. First-class functions allow Dart to do functional programming things like passing a function as an argument to another function. Functions can be stored in collections and also be returned from another function. Dart also provides support for anonymous functions, often known as lambda’s [13]. JavaScript also has first-class functions and is what enables the ability to make callbacks. This is a major difference between a language like Dart and Java. Java does not have first-class functions and consequently functions cannot be passed as a parameter to another function. The inability to provide some notion of a callback makes modern UI development difficult in a language without first-class functions like Java. Dart also supports syntax nearly identical to JavaScripts ES6 arrow functions

helloWorld() => print('Hello World');

which is just syntactic sugar for { return expr; } [13].

Dart was designed specifically for building user interfaces and sending and receiving information with APIs is a large part of client-server interactions. Often various API requests have optional parameters and for this reason the same is true with Dart functions. Optional parameters are notated with brackets [] as shown below.

String postUpdate(String username, String msg, [String img])

To create similar behavior in Java a method would need to be overloaded. ES6 added default argument values in function declarations to JavaScript [14], functionally that Dart also provides. Important to note while talking about functions is that unlike JavaScript but similar to Java/C, Dart applications require a top-level main() function that takes in an optional list of arguments.

Scope

While talking about functions is a good time to talk about how Dart is scoped. Similar to C, “Dart is lexically scoped. Scopes may nest.”[7] Lexical or static scope means that the scope is determined at compile time. Different from C though, Dart supports nested functions.

Operators

Dart’s operators are like what you would find in Java and Python with a few added for the specific intent of helping with common interface demands and client-server interactions. One of these is null-aware operators which are largely inspired by C#’s null-coalescing operator [15]. These are nice syntactic sugar to cut down on the necessary code to check for a potentially null value, object, or expression. For example the below code both do the same thing.

// Short version using ?? null operator
String usersName(String name) => name ?? 'Guest';

// Long version using a traditional if-else statement
String usersName(String name) {
   if (name != null) {
       return name;
   } else {
       return 'Guest';
   }
}

Null operators can also be added to assignments, object access and method invocation.

  // null-aware assignment
 var hello = null;
 x ??= 'Hello World';
 print(hello); // prints Hello World

Dart also has something called a spread operator which makes expanding and consolidating collections frictionless. Spread operators were also added to JavaScript in 2015 with ES6 [14].

Below is an example of a null operator paired with a spread operator. This can be helpful in the event loop of a component or widget of an application.

var newEvents; // null list
var recordedEvents = ['event1',  'event2', ...?newEvents];

Control Structures

Dart has the standard flow control tools you would find in an imperative language such as if/else, for loops, while/do-whiles, case switches. Something specific to Dart is that conditional statements must contain a boolean value, this is different from JavaScript which allows for non-boolean values to be placed in conditional expressions. Dart also addresses a common issue with JavaScript closures in for loops where the index value is not captured [16]. The below example is from Dart’s language documentation [17].

var callbacks = [];
for (var i = 0; i < 2; i++) {
   callbacks.add(() => print(i));
}
callbacks.forEach((c) => c());

The equivalent code in JavaScript would print out 2 twice as the index value is not captured. In Dart this code example would print 0 and 1 [17].

Identifiers

Unlike Java, Dart does not have public, private, or protected keywords. Rather in Dart all identifiers are public by default. To make a variable private in Dart you start the name with an underscore [18]. This decision is one I like as preceding private instance variables with an underscore is the proper style I was taught at George Fox University. In Dart preceding with an underscore not only symbolically means an instance variable is private, it syntactically and semantically means privacy too. Even though a small thing, this is a great example of an intuitive syntactic decision to take what has become common software development practice and remove unnecessary code.

Classes

Classes are the backbone of object-oriented languages and that is the case for Dart. Simply put from Dart’s language guide, “every object is an instance of a class, and all classes descend from Object” [19]. Dart also uses what is called Mixin-inheritance. Mixins are classes whose methods can be used by other classes without them needing to be a child. Mixins solve what Robert C. Martin (a.k.a “Uncle Bob”) called the “Deadly Diamond of Death” which is an ambiguity that arises with multiple inheritance [20]. Mixins remove the ambiguity of which class to inherit from. Another argument for the use of Mixins is that they enable proper modularity and consequently motivate code reuse.

Class behavior in Dart is very similar to Java, however, there are some key differences, such as the already mentioned lack of public, private, and protected identifiers. Dart also does not have an interface keyword. This is because in Dart every class defines an interface. Unlike Java, Dart auto generates getter methods for all public instance variables. The same is true for setters if the variable is a non-final instance variable. All instance variables are public by default. As mentioned earlier to make an instance variable private the identifier needs to precede with an underscore. Getters need to be made for these private instance variables. Another difference between Java and Dart is that subclasses do not inherit constructors from their superclass, rather subclasses need their own constructor or will use a default constructor with no parameters.

Isolates

        Unlike C and Java, Dart is not multi-threaded. To provide multithreading capabilities without a number of the issues that arise with concurrency Dart has what they call Isolates, inspired by Erlang’s lightweight processes [8]. Isolates are essentially standalone processes that have their own memory heap and cannot access the memory of their parents or any other Isolate for that matter, which means that each Isolate is it’s own contained thing. Isolates can communicate by sending messages back and forth but there is no shared memory, eliminating all of the issues that arise when sharing data concurrently. Isolates are always running in an event loop, processing events one by one in a queue fashion as they come in as. All Dart code is run within Isolates. This is partially helpful when an UI application needs to create and then destroy several components quickly. Memory allocation and garbage collection do not require any locking which means that they do not need to wait on the main thread or worry about any data that might have been changed [15].

Asynchronous

Asynchronous programming is very common when working with client-server interactions. Often to the frustration of a new JavaScript developer when they find their code continuing to execute while their just made API call has not returned yet! JavaScript does this because it is an event-driven language. The common solution in JavaScript is the use of callback functions, ensuring that the following code will not execute until the current function returns. Dart provides callback functionality but also has a Future object and async/await keywords to do the job. A Future is just like a JavaScript Promise in that it promises some result in the future and can be in one of three states: pending, completed or rejected. Promise, Futures, and async/await functionality greatly improve the ability to handle UI state changes when receiving data from a server. Earlier on in Dart’s life this could have been seen as a large developmental advantage as Promises were not added to JavaScript until ES6 in 2015 [14] and async/await in ES8 in 2017 [21]. Dart also has Stream objects that allow it to listen for events much like you would in JavaScript. Handling event listening with Isolates and Streams gives event-driven functionality without making it the entire language paradigm, something that would otherwise be detrimental to large scale projects.

Syntax

Dart falls within the C-style object-oriented language paradigm which is largely the reason the syntax looks the way it does, however, Dart looks very similar to JavaScript compared to C or Java. This was an intentional design decision to allow for efficient compilation to JavaScript [22]. Dart’s syntax is familiar, simple, intuitive and designed for productivity. When asked why Dart’s syntax is not more exciting Dart’s online documentation says that “Dart chooses familiarity over excitement” and one of Google’s developers said, “I wish it had a little more razzle dazzle but I can’t deny that literally on my first day of writing Dart code, I was productive in it” [22]. Something similar to Java but different is Dart’s use of comments. Normal comments giving clarity to what code is doing are the traditional double slash // just as in C and Java, however, Dart uses triple slash /// comments for markdown that gets converted to their dartdocs, similar to C#’s use of the /// for XML documentation. 

Compilation

Dart code is compiled using different strategies depending on the project goal and platform target either for native or web contexts. For native contexts Dart uses optimized AOT (ahead-of-time) compilation to produce ARM or x64 machine code. Dart code can also run in their VM which uses JIT (just-in-time compilation) and is designed to improve the development workflow with fast iterative development [23]. Development builds in the Dart VM have incredibly fast hot-reloading that seems instant. Hot-reloading is one of the most important tools for a productive workflow when building any type of visual interface. For web contexts, Dart transpiles directly to JavaScript with both a hot-reloading development build and an optimized production build that minifies and tree shakes the code [23].

My Experience

        One of Google’s design goals for the language was to make it “feel familiar and natural to programmers and thus easy to learn” [14] and I can say that this was my experience. Almost immediately I felt productive and competent with the language. I do a lot of web application development and have a decent amount of JavaScript and web framework experience. This prior experience I believe is a big reason Dart was so immediately familiar. The language behavior is intuitive, pragmatic, and unsurprising. The syntactic decisions made all lead the developer to intuitively see what is happening. An example of this is collection access with [ ] brackets. For example a list can be accessed list[index] and similarly maps with map[key]. Little decisions like this greatly improve the consistency of the language and consequently its writability. The standardization of similar behaving syntactic elements, such as item access, creates a rule of expectation for the developer that when you come across a collection and need to get an element there are no surprises.

Getting Dart up and running was incredibly easy. Google provides an online editor called DartPad that is just like Codepen, however, I downloaded the SDK onto my laptop and used Intellij. The language accessibility could not be any easier. The three programs that we coded did not touch on what Dart was designed for, building user interfaces, however, Dart was incredibly easy to work with to solve the problems. Dart’s tooling for building interfaces is incredible though. I was impressed with how instant the hot-reloading was. Flutter, the UI framework for Dart was also really nice. Everything in my opinion is in place to develop top-tier user experiences within applications without having to fight the technology and tools in place.

Dart’s Success as a Language

With Dart Google sought to create a structured yet flexible language for web programming that is familiar and intuitive and that is high performing in several different environments. In my opinion Google found a great middle point between flexibility and structure creating the least amount of friction during development. This is highlighted in the simple and intuitive syntactic and semantic elements. Especially for developers coming from a JavaScript background Dart is both a highly readable and writable language. Every decision in the language is intuitive and unto the intended task. Dart solves several issues JavaScript unintentionally creates for large scale application development. All of the technological features, tooling, and support are present and I believe Google achieved their ambitious goals they set out to accomplish. Google’s applications and websites have some of the best and most consistent user experiences across platforms which in my opinion alone speak volumes at Dart’s effectiveness. Their documentation on material design is largely considered canon in today’s UI design for application and website development. Pairing that with a language designed and optimized for building modern multi-platform applications with a single code-base, what more could you ask for?

void main(List<String> arguments) {
   /// The string input from the command-line
   var input;

   if (arguments.isNotEmpty) {
       input = arguments[0];

       if (isInt(input)) {
           print(factorial(int.parse(input)));
       } else {
           printErrorMessage('an integer was not provided!');
       }
   } else {
       printErrorMessage('no input was provided!');
   }
}

/// Recursively finds factorial of [n]
int factorial(int n) {
   if (n <= 1) {
       return 1;
   } else {
       return n * factorial(n - 1);
   }
}

/// Checks if the given string [s] is an int
bool isInt(String s) {
   if (s == null) {
       return false;
   }
   return int.tryParse(s) != null;
}

/// Prints the given error message [reason]
void printErrorMessage(String reason) {
   print('Error: $reason');
}

void main(List<String> arguments) {

   Grammar grammar;
   var dataFile;
   var start;
   var map;
   var queue;
   var random;

   if (arguments.isNotEmpty) {
       try {
           dataFile = File(arguments[0]);
           grammar = Grammar(dataFile.readAsStringSync());
       } catch (e) {
           print(e);
       }

       start = grammar.startSymbol;
       map = grammar.symbolTable;
       queue = Queue<Symbol>();
       random = Random();

       queue.addFirst(start);

       while(queue.isNotEmpty) {

           var token = queue.removeFirst();

           if (token.isNonTerminal()) {

               List<List<Symbol>> outerList = map[token];

               var innerList = outerList[random.nextInt(outerList.length)];

               for (var i = innerList.length - 1; i >= 0; i--) {
                   queue.addFirst(innerList[i]);
               }
           }
           else {
               stdout.write(token._value + ' ');
           }
       }
   }
   else {
       print('Error: No file was provided!');
   }
}

class Grammar {

   Symbol startSymbol;
   var symbolTable;

   Grammar(String input) {

       final START_TOKEN = '{';
       final END_TOKEN = '}';
       final BODY_END = ';';

       symbolTable = <Symbol, List<List<Symbol>>>{};

       List<Symbol> symbolList = [];

       var readingAProduction = false;
       var readingBodies = false;

       Symbol leftHandSideSymbol;
       String currentSymbol;

       var tokenizedInput = [];

       tokenizedInput = input.split(RegExp('\\s+'));

       var itr = tokenizedInput.iterator;

       while(itr.moveNext()) {
           currentSymbol = itr.current;

           if (!readingAProduction) {
               readingAProduction = (currentSymbol == START_TOKEN);
           }
           else if (currentSymbol == END_TOKEN) {
               readingAProduction = false;
               readingBodies = false;
           }
           else {

               if(!readingBodies) {
                   leftHandSideSymbol = Symbol((currentSymbol));

                   symbolTable[leftHandSideSymbol] = <List<Symbol>>[];

                   readingBodies = true;

                   // if startSymbol is not null then set it to leftHandSideSymbol
                   startSymbol ??= leftHandSideSymbol;
               }
               else {
                   if (currentSymbol == BODY_END) {
                       symbolTable[leftHandSideSymbol].add(symbolList);

                       symbolList = <Symbol>[];
                   }
                   else {
                       if (currentSymbol == '\\n') {
                           symbolList.add(Symbol('\n'));
                       }
                       else {
                           symbolList.add(Symbol(currentSymbol));
                       }
                   }
               }
           }
       }
   }
}

/// Simple class representing terminal and non-terminal grammar symbols
class Symbol {

   final RegExp NON_TERM = RegExp('^<.*>\$');

   final String _value;

   Symbol(this._value);

   bool isNonTerminal() {
       // See if the pattern matches the NON_TERM pattern
       return NON_TERM.hasMatch(_value);
   }

   bool isTerminal() {
       return !isNonTerminal();
   }

   @override
   bool operator ==(Object other) =>
       identical(this, other) ||
       other is Symbol &&
       runtimeType == other.runtimeType &&
       _value == other._value;

   @override
   int get hashCode => _value.hashCode;
}
